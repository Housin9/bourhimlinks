# Bourhim Links - Agent Instructions

Comprehensive coding standards and guidelines for LLM assistants working on the Bourhim Links project.

## Quick Reference

- **Project Type**: Next.js 16.2.1 (App Router) with TypeScript 5 (strict mode)
- **Database**: Drizzle ORM with Neon serverless (PostgreSQL)
- **UI Framework**: React 19 + Tailwind CSS 4 + shadcn/ui
- **Authentication**: Clerk
- **Language**: TypeScript (strict mode enabled)
- **Linting**: ESLint 9 with Next.js configuration

## Documentation Structure

This documentation is organized into domain-specific guides. **Read the relevant guide before writing code in that domain.**

ALWAYS refer to the relevant .md file BEFORE generating any code:

## When Working on This Project

### Before Writing Code

1. **Identify the domain** of the task (React component? API route? Database query? Styling?)
2. **Read the relevant guide** from the documentation structure above
3. **Review existing code** in that domain for patterns and consistency
4. **Ask clarifying questions** if requirements are ambiguous

### During Implementation

- **Follow conventions** from the relevant guide strictly
- **Match existing patterns** found in the codebase
- **Run `npm run lint`** before considering code complete
- **Handle errors explicitly** - no silent failures
- **Write meaningful comments** explaining *why*, not *what*
- **Test edge cases** mentally before implementing

### Code Quality Checklist

Before submitting code, verify:

- [ ] Follows naming conventions for the code domain
- [ ] Has appropriate TypeScript types (no `any` types)
- [ ] Includes error handling with meaningful messages
- [ ] Has comments explaining non-obvious logic
- [ ] All functions have explicit return types
- [ ] Props interfaces are defined for components
- [ ] No hardcoded values (use constants instead)
- [ ] Passes ESLint checks (`npm run lint`)
- [ ] Follows component/function organization patterns
- [ ] Dark mode support for UI components (if applicable)
- [ ] Proper imports organized by priority
- [ ] No console.log in production code (use logging utilities)

## Key Principles

### 1. Type Safety First

- Use TypeScript strict mode effectively
- Avoid `any` types - use `unknown` with proper narrowing
- Explicit return types for all functions
- Interfaces for component props

### 2. Consistency

- Follow existing patterns in the codebase
- Don't introduce new patterns without discussion
- Maintain consistent naming across domains
- Use established directory structure

### 3. Clarity

- Code should be self-documenting
- Comments explain *why*, not *what*
- Meaningful variable and function names
- Extract complex logic into separate, named functions

### 4. Performance

- Lazy load components when appropriate
- Avoid N+1 queries in database access
- Use `next/image` for all images
- Memoize only when necessary (avoid premature optimization)

### 5. Accessibility

- Color contrast passes WCAG standards
- Proper focus states visible
- ARIA attributes where appropriate
- Keyboard navigation support

## Common Tasks

### Adding a New Component

1. Determine if it's a UI component (`components/ui`) or feature component (`components/`)
2. Reference [React & Next.js](./react-next.md) for component structure
3. Reference [Styling & UI](./styling.md) for Tailwind patterns
4. Define prop types with TypeScript interfaces
5. Include dark mode support
6. Export from `index.ts` file in the directory

### Creating an API Endpoint

1. Reference [API & Backend](./api.md) for route structure
2. Follow the naming convention for file paths
3. Include request validation
4. Handle authentication if needed
5. Return proper HTTP status codes
6. Add comprehensive JSDoc comments

### Adding Database Functionality

1. Reference [Database & ORM](./database.md)
2. Update schema in `db/schema.ts` if needed
3. Create query functions in `db/queries.ts`
4. Use proper transaction handling for related operations
5. Run migrations after schema changes

### Fixing a Bug

1. Understand what the code *should* do
2. Trace through the code path systematically
3. Check TypeScript types - many bugs are type mismatches
4. Consider database queries - N+1 is common
5. Verify error handling - is the error being caught correctly?
6. Look for similar patterns in the codebase that work

## Important Notes

### Next.js 16 Differences

This project uses **Next.js 16.2.1** - verify breaking changes in `node_modules/next/dist/docs/` before assuming standard patterns.

### TypeScript Strict Mode

Strict mode is **enabled**. This means:
- `any` types are not allowed
- Null/undefined must be handled explicitly
- All function parameters must be typed
- All function return types must be explicit

### Tailwind CSS 4

This project uses **Tailwind CSS 4**, which has breaking changes from v3. Verify class names in the project before assuming standard patterns.

### Clerk Authentication

Authentication is handled by Clerk. All user auth operations go through Clerk, not custom solutions.

## File Naming Quick Reference

| Type | Convention | Example |
|------|-----------|---------|
| Components | PascalCase | `UserCard.tsx` |
| Functions/Utils | camelCase | `formatDate.ts` |
| Hooks | camelCase + `use` | `useAuth.ts` |
| Constants | UPPER_SNAKE_CASE | `API_BASE_URL.ts` |
| Tables | snake_case | `users`, `user_profiles` |
| Columns | snake_case | `user_id`, `created_at` |
| Directories | kebab-case (generally) | `components/ui/` |
